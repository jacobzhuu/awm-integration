# Unified Verification Interface

## Objective

Define one result contract that can represent three verification evidence chains without forcing them into the same underlying metric system:

- AWM: white-box lineage verification
- Model-Fingerprint Adapter: gray-box ownership verification
- watermark: output-side verification

The interface should let downstream systems consume all three backends through one stable envelope while still preserving backend-specific evidence.

## Evidence Chains

### AWM

- evidence type: white-box
- primary input: local checkpoint weights
- main signal: weight-space similarity after LAP alignment and CKA
- current local decision rule: z-score thresholds
- best fit: lineage or relatedness verification between open checkpoints

### Model-Fingerprint Adapter

- evidence type: gray-box
- primary input: model plus ownership-side adapter evidence
- main signal: adapter-specific ownership score or confidence
- best fit: ownership verification when full white-box lineage evidence is unavailable or insufficient

### Watermark

- evidence type: output-side
- primary input: generated text, decoding metadata, or watermark key context
- main signal: watermark detector score or p-value
- best fit: proving that an output stream was generated by a marked model family

## Unified Return Fields

Every backend should return the same top-level fields.

```json
{
  "tool": "awm",
  "backend": "white_box",
  "status": "success",
  "decision": "related",
  "decision_reason": "absolute_z_score >= 2.0",
  "started_at": "2026-04-15T00:00:00Z",
  "finished_at": "2026-04-15T00:01:00Z",
  "subject": {},
  "reference": {},
  "artifacts": {
    "log_path": "",
    "result_path": ""
  },
  "metrics": {},
  "thresholds": {},
  "evidence": {
    "mode": "white_box",
    "summary": ""
  },
  "error": null
}
```

## Field Semantics

- `tool`: concrete backend identifier such as `awm`, `model_fingerprint_adapter`, or `watermark`
- `backend`: coarse backend family such as `white_box`, `gray_box`, or `output_side`
- `status`: execution health, usually `success` or `failed`
- `decision`: thresholded backend verdict
- `decision_reason`: short human-readable explanation of the thresholding result
- `started_at` and `finished_at`: UTC timestamps for run tracing
- `subject`: the item being verified
- `reference`: optional comparison or provenance target
- `artifacts.log_path`: raw log emitted by the underlying tool
- `artifacts.result_path`: structured backend result written to disk
- `metrics`: raw backend metrics without cross-backend coercion
- `thresholds`: explicit threshold rules used to derive `decision`
- `evidence.mode`: `white_box`, `gray_box`, or `output_side`
- `evidence.summary`: short backend-specific explanation
- `error`: null on success, otherwise a failure message

## Backend-Specific Metric Examples

### AWM metrics

```json
{
  "average_wq_weights_pct": 100.0,
  "average_wk_weights_pct": 100.0,
  "average_wq_wk_weights_pct": 100.0,
  "reference_similarity_pct": 0.38,
  "absolute_z_score": 355.8
}
```

### Model-Fingerprint Adapter metrics

```json
{
  "backend_score": 0.91,
  "backend_confidence": 0.88
}
```

### Watermark metrics

```json
{
  "detector_score": 12.4,
  "p_value": 0.0003,
  "tokens_evaluated": 512
}
```

## Decision Vocabulary

Decision values should remain backend-specific but interpretable:

- AWM: `related`, `uncertain`, `not_related`
- Model-Fingerprint Adapter: `supported`, `uncertain`, `not_supported`
- watermark: `watermarked`, `uncertain`, `not_watermarked`

This keeps the semantics faithful to each backend while preserving one shared structure.

## Integration Guidance

To avoid over-coupling:

- keep each backend in its own wrapper script
- never force all backends to emit the same metric names
- do require the same top-level envelope and artifact bookkeeping
- keep threshold rules explicit in the result object
- let downstream orchestration decide how to combine or rank multiple backend decisions

## Recommended Aggregation Layer

A future orchestrator can consume backend outputs as a list:

```json
{
  "verification_id": "case_001",
  "results": [
    {"tool": "awm", "decision": "related"},
    {"tool": "model_fingerprint_adapter", "decision": "supported"},
    {"tool": "watermark", "decision": "not_watermarked"}
  ]
}
```

This keeps the integration layer simple and avoids pretending that white-box, gray-box, and output-side evidence are numerically interchangeable.
